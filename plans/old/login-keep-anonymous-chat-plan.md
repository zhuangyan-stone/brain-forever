# 匿名用户登录后清除对话历史的 UX 分析与优化方案

## 一、当前逻辑还原

### 完整用户流程

```
匿名访客 ──> 开始对话（发送消息，AI回复）──> 点击登录 ──> 调用 POST /api/chat/login ──> 切换为登录用户
```

### 后端做了什么

核心代码在 [`internal/agent/types.go:219-245`](../internal/agent/types.go:219) 的 `switchToUser()` 方法：

1. 打开用户数据库 `data/{sn}.chats.db`
2. 从数据库加载用户的会话列表（最近 100 条）
3. ⚠️ **将 `currentChat` 设为 `nil`** — 清空匿名会话的内存历史
4. 设置 `chatStore`（数据库存储对象）和 `chats`（会话列表）
5. [注释原文](on_login.go:40): `// Switch the session to the logged-in user`

### 前端做了什么

[`chat-api.js:187-259`](../frontend/static/chat-api.js:187) 的 `switchToUser()`：

1. 清空所有消息状态 `state.messages = []`
2. 移除所有消息 DOM 节点
3. 移除欢迎消息
4. 更新登录按钮文本
5. 重新加载会话列表（来自后端返回的登录后数据）
6. 显示欢迎消息（空状态）

### 结果

用户登录后，**之前匿名聊天的所有内容完全消失**，页面回到初始欢迎状态，感觉像"什么都没发生过"。

### 为什么匿名聊天不能自动保留

- 匿名用户的 [`chatStore == nil`](../internal/agent/db.go:35)，意味着**没有任何数据库持久化**
- 匿名会话只在内存中的 `currentChat.history` 存在
- 登录时 `switchToUser` 将 `currentChat = nil`，内存数据自然丢失
- 没有将匿名聊天的消息写入用户数据库的逻辑

## 二、UX 分析：这个设计是对是错？

### ❌ 从用户体验视角：**这是有问题的设计**

| 维度 | 问题 |
|------|------|
| **连续性断裂** | 用户已经和 AI 进行了有意义的对话，积累了上下文。登录后一切归零，对话语境完全丢失 |
| **困惑感** | 用户会疑惑"我刚才的对话去哪了？"——没有任何提示说明为什么消失 |
| **信任感伤害** | 用户可能会认为系统不稳定或数据丢失，降低对产品的信任 |
| **工作效率损失** | 如果匿名时输入了长消息或重要的对话内容，这些工作成果永久丢失 |
| **认知失调** | 大多数 SaaS 产品的登录行为是"合并/保留已有数据"，而非"清空" |
| **未提供选择** | 用户没有被告知"登录后匿名对话将被清除"，也没有选择权 |

### ✅ 从技术/隐私视角：**也有合理考量**

| 维度 | 说明 |
|------|------|
| **隐私设计** | 匿名会话不关联用户身份，是一种隐私保护机制 |
| **数据归属** | 匿名数据无法确定属于哪个用户，强行关联可能有合规风险 |
| **实现简单性** | 两种存储后端不同（内存 vs SQLite），合并需要额外逻辑 |

### 结论

**从用户体验为核心的产品思维来看，当前设计是错误的。** 技术实现上的便利不应以牺牲用户体感为代价。但直接"粗暴合并"也不是最佳方案，需要做更精细的设计。

## 三、优化方案（推荐程度从高到低排列）

### 方案 A（推荐 🌟）：静默合并 + 新对话标记

**核心思路**：登录时自动将匿名对话持久化到用户数据库，并作为用户的一个历史会话。

**后端改动**：
1. `switchToUser()` 中，在 `currentChat = nil` **之前**，先判断是否有匿名聊天记录
2. 如果有（`currentChat != nil && len(currentChat.history) > 0`）：
   - 调用 `ensureDBSession()` 在用户数据库创建会话记录
   - 遍历 `currentChat.history`，调用 `persistMessageToDB()` 持久化每条消息
3. 然后再执行原有的切换逻辑

**前端改动**（最小化）：
- 登录后，如果后端返回了合并后的会话列表，新合并的匿名会话出现在列表顶部，标题为截取的第一个用户消息
- 登录后页面停留在"刚刚匿名聊天"的恢复状态，而非欢迎页面

**优点**：
- 用户无感知，对话不会丢失
- 技术实现简单，复用已有 `ensureDBSession` + `persistMessageToDB`
- 用户登录后看到的是刚刚的聊天记录，体验自然连贯

**缺点**：
- 如果同一浏览器之前有其他人匿名使用，数据会关联到登录用户（但 session cookie 已绑定到当前浏览器）

---

### 方案 B（次选 ✅）：弹窗询问用户

**核心思路**：登录时弹出对话框，询问用户是否要保存匿名对话。

**流程**：
```
用户点击登录 ──> 弹窗："您有一段匿名对话，是否保存到您的账户？"
                ├── [保存并继续] ──> 执行方案 A 的合并逻辑
                └── [不保存，开始新对话] ──> 执行现有清除逻辑
```

**优点**：
- 尊重用户选择权，隐私透明
- UX 友好，用户有掌控感

**缺点**：
- 增加了交互步骤，略繁琐
- 需要前端新增对话框组件和后端条件分支

---

### 方案 C（最激进 🚀）：登录前强制保存

**核心思路**：将"匿名→登录"的门槛降低，让用户在登录**前**就知道他们的数据会被保留。

**具体做法**：
- 匿名用户的聊天也**写入本地数据库**（启用匿名用户的 `chatStore`）
- 但匿名用户的数据写在**临时/公共数据库中**
- 登录时：从匿名数据库读取数据，迁移到用户专属数据库，删除匿名数据

**优点**：
- 匿名用户体验更稳定（即使刷新页面也不会丢失对话）
- 数据迁移透明

**缺点**：
- 架构改动大，需要同时维护匿名和登录两套数据库
- 复杂度显著增加

## 四、推荐实施步骤（方案 A）

### Step 1: 修改 [`switchToUser`](../internal/agent/types.go:219) 

在 `s.currentChat = nil` 之前，增加匿名聊天持久化逻辑：

```go
func (s *session) switchToUser(sn string) {
    // Phase 0: 保存匿名聊天到用户数据库（如果有匿名历史）
    s.mu.Lock()
    hasAnonymousHistory := s.currentChat != nil && len(s.currentChat.history) > 0
    if hasAnonymousHistory {
        anonymousHistory := s.copyHistoryWithoutLock()
        anonymousTitle, anonymousTitleState := s.getTitleWithoutLock()
    }
    s.mu.Unlock()

    // Phase 1: IO operations (no lock needed — DB creation + query)
    dbFile := "data/" + sn + ".chats.db"
    chatStore, err := store.CreateLocalChatScheme(dbFile)
    // ... 原有逻辑 ...

    // Phase 1.5: 如果有匿名历史，持久化到新数据库
    if hasAnonymousHistory {
        // 1. 创建新会话记录
        // 2. 持久化所有消息
        // 3. 添加到 chats 列表
    }

    // Phase 2: 原有切换逻辑
    s.chatsMu.Lock()
    s.chatStore = chatStore
    s.chats = chats
    s.chatsMu.Unlock()

    s.mu.Lock()
    s.currentChat = nil // 清空（数据已持久化）
    s.userNo = sn
    s.mu.Unlock()
}
```

### Step 2: 修改 [`OnLogin`](../internal/agent/on_login.go:20)

登录成功后，将新建的匿名持久化会话信息返回到前端：

```json
{
    "status": "ok",
    "user_no": "test_user_001",
    "sessions": [...],               // 包含新合并的匿名会话
    "restore_sn": "chat-xxx-123"     // 新增：指示前端要恢复哪一个会话
}
```

### Step 3: 修改前端 [`switchToUser`](../frontend/static/chat-api.js:187)

如果后端返回了 `restore_sn`，则调用会话恢复接口加载该会话的消息，而不是显示空欢迎页。

### Step 4: 处理边缘情况

- **匿名无对话**：没有聊天记录 → 执行现有逻辑，直接显示欢迎页
- **匿名对话已被删除**（用户调用了删除消息 API）→ 同上
- **数据库创建失败**：回退到清除逻辑，显示欢迎页 + 错误提示

## 五、副作用与风险

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| **数据重复** | 匿名聊天的消息 ID 和用户已有的 ID 可能冲突 | 使用 `ensureDBSession` 创建设置新的 `dbSessionID`，消息 ID 沿用原有 ID（在 user session 范围内是唯一的） |
| **性能影响** | 大量匿名聊天消息需要写入 DB，可能增加登录延迟 | 单个匿名会话通常不会超过上百条消息，SQLite 批量写入很快；如担心可放 goroutine 异步执行 |
| **并发安全** | 登录时可能正在流式输出 (streaming) | 前端登录前已 `abort()` 中止流式输出，后端持 `s.mu` 时不会有并发写入 |
| **session cookie 复用** | 同一浏览器不同用户使用 | 这是 session cookie 本身的限制，与本次改动无关 |

## 六、总结

| 项目 | 内容 |
|------|------|
| **当前设计评价** | ❌ 从 UX 角度是错误的——匿名对话无故消失，用户困惑 |
| **推荐方案** | 🅰️ 静默合并：登录时自动将匿名对话持久化到用户账户 |
| **实施难度** | 低-中，核心代码改动集中在 [`types.go`](../internal/agent/types.go) 一个文件 |
| **用户体感提升** | 显著——"登录后对话还在"是最基本的产品期望 |
