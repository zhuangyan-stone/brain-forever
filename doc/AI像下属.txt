1、他非常容易“轻信”以及“屈服”你
	有时候，你只是因为不理解其做法，而问一句“为什么这里需要……” ，或 “为什么你会……”，它都会优先当作是你在反对。

2、多数时候，不敢“冒险”，会走最保险的实现，特别是，现有的实现可以复用时。比如：修改列表中某个元素的标题，会选择重新渲染整个列表。不敢冒险本质是“不敢做会引发变化的决定”，因此还会表现为：做事方法的颗粒度会比较小。不敢将多个操作组合起在必要时，来变成一个较大的操作。（原因是：细颗粒度更精装地体现了事情的实现过程）

     不管冒险还表现在： 总是会非常冗余地写各种条件，或者非常不优雅地“防御性”编程。典型原因如：
     因为  原因 A，造成  问题B；但是，修改时，它会多写不少直接判断，以实现所谓“防御性”地杜绝问题B再出现……
    实例 ： 某类对象，使用 SN 来实现唯一性判断，但同时它也有 ID （比如数据库自增ID）成员。编程过程中，由于生成SN的代码有问题，造成重复SN。而后， AI 在分析过程中，会先有个中间思考 ：“二者SN一样，但是ID是不一样的啊？所以，是不是我不应该只用  SN 做去重，我应该把 ID也加上就万无一失了……” 。结果是：哪怕它最后查出“根因”是 SN 生成不对，但也会在代码保留下面的“保险”代码：
    if (a.SN != b.SN && a.ID != b.ID ) ..... // 容易误导读者 
   

3. 指挥者要有一定“容错度”（不要当一个有“现时完美主义者”的领导），即，如果不要急于纠正AI在实现某个既定小目标的过程中，看到错误（甚至就是不足），不要直接下命令要求（暂停当前操作）马上订正。
   简页言之，领导要有 “一件事一件事来的”的包容力（领袖气质）
	
   你知道它是错的，并且清楚这个错误的本质，清楚这个错误的影响范围，你要容错。但是，当你并不明白它在做什么时，你必须宁可让它停止，“不耻下问” 地了解对方的思考。

4. 及时的，小步的重构，越来越重要！

5. AI有时候为了达到目的（完成任务），会有很疯狂的做法！！ 。比如：让查找代码中所有的汉字时，它在简单地使用编码值区域失败（方法方向是对的，只是具体实现有bug）后，就开始疯了地使用“穷举法”，试图列出软件系统常见的用语、用字，然后全局查换替换……

6. 当功能一个个增加，实现之后，代码可能可以进行什么样的优化，基本上就得你先知道。哪怕最简单的：要不要新建多个文件，将相同的功能统一归于此处。

7. 你的测试队伍一定要增强——当然，和测试队伍引入AI（从而可能还减员了）并不矛盾。

8. 也许你确实不存在某个下属真正全能：什么都懂，什么都能干上活……但AI时代，你首先要理解：AI虽然看起来“什么都懂”，但它仍然有一件事不懂：不懂如何妥当地切换自己的角色。——让AI在合适的时机切换合适的角色，是你的责任。
